# Dependencies 層

## 責務
Dependencies層は、アプリケーションで使用する依存関係を組み立て、各レイヤーへ注入するComposition Rootとしての役割を持つ。

- 設定値を読み込む
- DB接続、HTTPクライアント、SDKクライアントなどを生成する
- Infrastructure層の実装を生成する
- Application層またはDomain層で定義されたProtocolとInfrastructure層の実装を結び付ける
- ユースケースへ必要な依存関係を注入して生成する
- 生成した具象ユースケースを入力ポート型としてPresentation層へ提供する依存性注入関数を定義する
- 外部クライアントや接続などのライフサイクルを管理する

Dependencies層は依存関係を組み立てるだけであり、各レイヤーの責務を代行しない。

## 依存可能なレイヤー
- Domain層
- Application層
- Infrastructure層
- FastAPIなど、Presentation層で使用する依存性注入フレームワーク

Dependencies層は、具体的な実装と抽象を結び付けるため、Domain層、Application層、Infrastructure層を参照してよい。RouterやSchemaなど、Presentation層のモジュールには依存しない。

Domain層、Application層、Infrastructure層からDependencies層へ依存させてはならない。Presentation層は、ユースケースなどを受け取るためにDependencies層の依存性注入関数を参照してよい。

## 配置するもの
- 設定値を読み込む処理
- 外部クライアントを生成する処理
- Infrastructure実装を生成する処理
- ユースケースを生成する処理
- FastAPIの`Depends`で利用する依存性注入関数
- アプリケーション起動時および終了時のリソース管理

## 配置しないもの
- 業務ロジック
- 業務上の認可判断
- ユースケースの処理手順
- リクエスト値からApplication入力への変換
- Application出力からレスポンスへの変換
- Domain例外やApplication例外からHTTPレスポンスへの変換
- Repositoryや外部APIクライアント自体の実装
- Repository操作の呼び出し

## 依存関係の組み立て
依存関係は、外側の具体的な実装から内側のユースケースへ向けて組み立てる。

```text
設定値・外部クライアント
  → Infrastructure実装
  → Application / DomainのProtocol
  → ユースケース
  → Presentation層
```

ユースケースやInfrastructure実装は、自身が必要とする依存関係をコンストラクタ引数として受け取る。実装内部でDependencies層の関数を直接呼び出して依存関係を取得してはならない。

依存性注入関数は、必要な依存関係を生成または取得して返すことだけを行う。リクエスト内容や業務状態によって、利用する実装を切り替える判断は行わない。

環境や設定によって利用する実装を切り替える場合は、Dependencies層で明示的に選択する。

## Docstring
Dependencies層では、DIファクトリが何を返すかを短く示す。

- 公開DIファクトリには一言のdocstringを書く
- 名前だけで完全に自明なprivateヘルパーにはdocstringを書かなくてよい
- 外部クライアント生成、キャッシュ、環境変数依存など、呼び出し側に影響する事情がある場合は概要に含める
- Args、Returns、Raisesは原則として省略する

docstringはGoogle Style Docstringを使用する。詳細な生成手順や設定値の意味は、対象のFactoryやSettings側へ寄せる。

## スコープとライフサイクル
依存関係ごとに適切な共有範囲とライフサイクルを定める。

- 設定値など不変のオブジェクトは、アプリケーション単位で共有してよい
- コネクションプール、HTTPクライアント、SDKクライアントなど共有可能なクライアントは、原則としてアプリケーション単位で生成する
- DBセッションなどリクエストごとに分離すべきリソースは、リクエスト単位で生成する
- 終了処理が必要なリソースは、利用終了時またはアプリケーション終了時に確実に破棄する
- Infrastructure実装およびユースケースの共有範囲は、注入する依存関係のライフサイクルに合わせる

具体的な生成・破棄方法は、使用するフレームワークやSDKに応じて定義する。

## トランザクション
Dependencies層は、Application層が定義する1ユースケース1トランザクションの方針を実現するために必要な依存関係を組み立てる。

ただし、Dependencies層自身が業務処理の成功・失敗を判断して、独自にトランザクション境界を変更してはならない。具体的なトランザクション制御方式は、実装時にApplication層およびInfrastructure層の方針と整合させて決定する。

## 設定・機密情報
- APIキー、DB接続情報、AWS認証情報などをコードへ直接記述しない
- 設定値は環境変数またはSecrets Managerなどから読み込む
- 読み込んだ設定値は、必要なInfrastructure実装へ明示的に注入する
- 機密情報を例外や外部公開レスポンスへ含めない
- 必須設定が不足または不正な場合は、アプリケーション起動時に検出して起動を失敗させる

## エラーハンドリング
Dependencies層は、Domain例外、Application例外、Infrastructure層が出力ポートの契約へ変換した例外を捕捉・変換しない。

依存関係の生成や設定値の読み込みに失敗し、アプリケーションを正常に動作させられない場合は、例外を握り潰さず起動を失敗させる。

リソースの破棄処理で例外が発生した場合も握り潰さない。ただし、処理中に発生した元の例外を破棄処理の例外で隠さないようにする。

## テスト方針
- Protocolと意図したInfrastructure実装が正しく結び付けられることをテストする
- ユースケースへ必要な依存関係が正しく注入されることをテストする
- 設定に応じて意図した実装が選択されることをテストする
- 必須設定が不足または不正な場合に、依存関係の生成が失敗することをテストする
- リクエスト単位およびアプリケーション単位のリソースが、定義されたスコープで生成・破棄されることをテストする
- テスト時に依存関係をモックまたはスタブへ差し替えられることを確認する
