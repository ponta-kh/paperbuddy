# Presentation 層

## 責務
Presentation層は、HTTPなどの外部インターフェースとApplication層の境界として、入出力の変換を行う。

- 外部からリクエストを受け取る
- リクエスト値の形式的な妥当性を検証する
- リクエスト値と認証結果をApplication層の入力へ変換する
- 依存性注入された入力ポートを通してユースケースを呼び出す
- ユースケースの戻り値をレスポンススキーマへ変換する
- Domain層およびApplication層から送出された例外を外部インターフェースのエラーへ変換する
- 使用する認証方式に従ってユーザーを認証する
- 認証要否に関わらず、リクエスト単位の識別子を採番する

業務ロジックや業務上の認可判断は行わない。

## 配置するもの
- Router
- リクエストスキーマ
- レスポンススキーマ
- 認証処理
- リクエスト単位の識別子を採番する依存性注入処理
- 例外からHTTPレスポンスへの変換処理

## 配置しないもの
- 業務ロジック
- 業務上の認可判断
- Repositoryや外部APIクライアントの直接呼び出し
- Domainオブジェクトの生成や状態変更
- DBトランザクション制御
- ユースケースの生成や依存関係の組み立て

## スキーマ
HTTPエンドポイントのリクエストとレスポンスには、Pydanticの`BaseModel`を使用する。

リクエストスキーマとレスポンススキーマは分けて定義し、原則としてエンドポイントごとに独自のスキーマを作成する。Application層の入力モデル、出力モデル、DomainオブジェクトをHTTPスキーマとして直接使用してはならない。

リクエストスキーマは`[処理名]Request`、レスポンススキーマは`[処理名]Response`と命名する。リクエストBodyが存在しない場合は、リクエストスキーマを作成しない。

リクエストBodyのスキーマでは、`ConfigDict(extra="forbid")`を指定する。

HTTPのパスパラメータやクエリパラメータは文字列として受信するため、Presentation層ではHTTP表現からAPIで定義した型への変換を許容する。`strict=True`は一律に指定せず、暗黙変換を禁止する必要があるフィールドに個別に指定する。

スキーマには、API利用者が公開仕様を理解するために必要なフィールドの説明、制約、例を定義する。

複数のエンドポイントで共通となるエラーレスポンスなどは、責務と形式が同一である場合に限り共有してよい。

### 検証するもの
- 型
- 必須項目
- 文字列長や数値範囲など、値単体で判定できる制約
- 複数のリクエスト項目だけで判定できる構造的な整合性

### 検証しないもの
- データベースや外部サービスへの問い合わせが必要な条件
- 認証済みユーザーが対象を操作または参照できるかという業務上の認可
- Domain層が保証すべき業務ルール

これらは、Application層またはDomain層で検証する。

## Router
Routerは薄く保ち、以下の処理のみを行う。

1. リクエスト値と認証結果を受け取る
2. Application層の入力を構築する
3. 依存性注入されたユースケースの`execute`を呼び出す
4. ユースケースの戻り値をレスポンススキーマへ変換して返す

```python
async def get_chat(
    chat_id: UUID,
    user: AuthenticatedUser,
    use_case: GetChatProtocol,
) -> GetChatResponse:
    query = GetChatInput(chat_id=chat_id, user_id=user.id)
    output = await use_case.execute(query)
    return GetChatResponse(chat_id=output.chat_id, message=output.message)
```

Router内でユースケースを生成してはならない。また、Router同士を呼び出してはならない。
Routerは具象ユースケースをimportせず、Application層の入力ポートProtocolへ依存する。

Application層の出力からレスポンススキーマへの変換は、Router内で直接行う。変換処理が複雑になった場合は、その時点でMapperなどへの分離を検討する。

## リクエストID
Presentation層は、FastAPIの依存性注入を使ってリクエストIDをUUID v7で採番する。

リクエストIDは、認証が必要なエンドポイントと不要なエンドポイントのどちらでも採番する。認証失敗でユースケースを呼び出さない場合でも、Presentation層で受け付けたリクエストとして同じ採番規則を適用する。

同一HTTPリクエスト内で複数回リクエストIDを参照する場合は、同じ値を使用する。ユースケース入力へ渡す必要がある場合は、HTTP表現ではなくApplication層の入力値として明示的に変換する。

## 認証・認可
HTTPヘッダー、Cookie、アクセストークンなど、使用する認証方式に従った認証処理はPresentation層で行う。

認証結果からユーザーIDなどのユースケース実行に必要な情報を取り出し、Application層の入力へ含めて渡す。

認証されていないリクエストは、ユースケースを呼び出さずに拒否する。認証済みユーザーが対象データを参照または操作できるかという業務上の認可は、Application層またはDomain層に委ねる。

## エラーハンドリング
Domain層およびApplication層から送出された想定可能な例外を、HTTPステータスコードとエラーレスポンスへ変換する。

例外とHTTPレスポンスの対応は、FastAPIのException Handlerを使用して一元管理する。Routerごとに同じ`try` / `except`を記述しない。

- 認証失敗は`401 Unauthorized`へ変換する
- 業務上の認可失敗は`403 Forbidden`へ変換する
- 想定可能なDomain例外およびApplication例外は、意味に対応するHTTPステータスコードへ変換する
- 予期しない例外は、内部情報を隠した`500 Internal Server Error`へ変換する

Domain例外およびApplication例外のメッセージを、そのままエラーレスポンスとして公開してはならない。

Exception Handlerは例外の型に応じて、HTTPステータスコード、エラーコード、利用者向けメッセージを決定する。利用者向けメッセージはPresentation層で定義し、Domain層およびApplication層に外部公開形式を意識させない。

例外メッセージ、スタックトレース、SDK固有のエラー内容、DB情報などの内部情報をレスポンスへ含めてはならない。

HTTPエラーレスポンスの形式は、[バックエンドAPI共通規約](../api-conventions.md)に従う。

## バッチ処理
バッチ処理の呼び出し口に関する規約は、必要になった時点で定義する。

## テスト方針
- リクエスト値の制約は、同値分割と境界値分析に基づいてテストする
- リクエスト値と認証結果がApplication層の入力へ正しく変換されることをテストする
- Application層の出力がレスポンススキーマへ正しく変換されることをテストする
- Domain例外およびApplication例外が、定義されたHTTPレスポンスへ変換されることをテストする
- 認証失敗時にユースケースが呼び出されないことをテストする
- ユースケースはモックまたはスタブに差し替え、Presentation層のテストから業務ロジックを分離する
