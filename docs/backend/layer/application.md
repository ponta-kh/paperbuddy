# Application 層

## 責務
Application層は、システムが提供するユースケースを実現する。

- Domain層のオブジェクトを利用して、ユースケースの処理手順を組み立てる
- Repositoryや外部サービスを出力ポート経由で呼び出す
- ユースケース固有の業務チェックを行う
- ユースケースの入力境界で、受け付ける値の構造的妥当性を保証する
- ユースケースの実行結果を、呼び出し元に依存しない形式で返す

HTTPステータスコードやリクエスト形式など、特定の呼び出し元に依存する処理は行わない。

## ユースケースの命名規約
- ユースケース名は「動詞 + 名詞」の順で命名する
- 引数の型は`[ユースケース名]Input`とする
- 戻り値の型は`[ユースケース名]Output`とする
- Queryユースケースの引数名は`query`、Commandユースケースの引数名は`command`とする

例えば、チャットを取得するユースケース名は`GetChat`とし、引数の型は`GetChatInput`、戻り値の型は`GetChatOutput`、引数名は`query`とする。

`input`はPythonの組み込み関数と重複するため、ユースケースの引数名には使用しない。

## QueryとCommandの判断基準
ユースケースの実行によって業務上の状態を永続的に変更するかどうかで、QueryとCommandを分類する。

- Queryは、業務上の状態を変更せず情報を取得するユースケースとする
- Commandは、登録・更新・削除などによって業務上の状態を永続的に変更するユースケースとする

認可やその他のドメイン判断を行う場合でも、業務上の状態を変更しなければQueryとする。

参照処理であっても、閲覧済み状態、最終アクセス日時、閲覧回数などを更新する場合はCommandとする。

ログ、メトリクス、トレースなどの運用上の記録は、業務上の状態変更には含めない。そのため、これらを記録するだけであればQueryのままとする。

## ユースケースの実行形式
- ユースケースはクラスとして定義する
- 1クラスは1ユースケースのみを担当する
- 実行メソッド名は`execute`とする
- 外部I/Oを伴うユースケースは`async`で定義する

```python
class GetChatUseCase:
    async def execute(self, query: GetChatInput) -> GetChatOutput:
        ...
```

## 入力モデル
ユースケースごとに独自の入力モデルを作成し、Pydanticの`BaseModel`を使用して定義する。入力モデルを複数のユースケースで共有してはならない。

Presentation層でもリクエスト値を検証するが、Application層はHTTP以外のバッチ処理やテストなどから直接呼び出される可能性がある。また、呼び出し元の検証漏れや誤りによって、不正な値が渡される可能性もある。

そのため、Application層は自身の入力境界で値を再検証し、ユースケースが不正な入力値を前提に処理を開始することを防ぐ。

Application層の入力モデルは、Presentation層のリクエストスキーマとは分けて定義する。Presentation層は外部から受け取った値をApplication層の入力モデルへ変換してから、ユースケースを呼び出す。

入力モデルでは、意図しない型変換や未定義フィールドの受け入れを防ぐため、原則として`ConfigDict(strict=True, extra="forbid")`を指定する。

ただし、引数が`str`型の値1つだけの場合など、独自の入力モデルを作成しても入力境界や意味が明確にならない場合は、その値を直接受け取ってよい。

### 検証するもの
- 型
- 必須項目
- 文字列長や数値範囲など、値単体で判定できる制約
- 複数の入力項目だけで判定できる構造的な整合性

### 検証しないもの
- データベースや外部サービスへの問い合わせが必要な条件
- ユーザーの権限や現在の状態によって結果が変わる条件
- EntityやValue Objectが保証すべきドメイン不変条件

これらは、ユースケースまたはDomain層で検証する。

## 出力モデル
ユースケースごとに独自の出力モデルを作成し、標準ライブラリの`dataclass`を使用して定義する。出力モデルを複数のユースケースで共有してはならない。

戻り値はApplication層で生成し、Presentation層がレスポンススキーマへ変換する。FastAPIやPydanticなど、Presentation層の都合には依存させない。

ただし、戻り値が`str`型の値1つだけの場合など、独自の出力モデルを作成しても戻り値の意味が明確にならない場合は、その値を直接返してよい。

## 出力ポート
出力ポートには、ユースケースが必要とする操作のみを定義する。Infrastructure層の実装に合わせた汎用的なCRUD操作を定義してはならない。

出力ポートの引数、戻り値、送出する可能性のある例外には、SDK、ORM、DBドライバーなどの実装固有の型を使用しない。

RepositoryのProtocolは、読み取りを行うQuery Repository Protocolと、状態変更を行うCommand Repository Protocolに分けて定義する。
※Infrastructure層のRepository実装を同様に分割するかどうかはInfrastructure層の都合で判断する。

## トランザクション境界
1ユースケースを1トランザクション境界とする。

Application層はユースケースの開始から正常終了までを1つの処理単位として扱い、正常終了時にコミットし、例外発生時にロールバックする。具体的なトランザクション実装には依存せず、出力ポートを通じて制御する。

## 認証・認可
ユーザーが誰であるかを確認する認証処理は、Presentation層の責務とする。Application層は、HTTPヘッダー、Cookie、アクセストークンなどの認証方式には依存しない。

Presentation層は、認証によって特定したユーザーIDなど、ユースケースの実行に必要な認証結果を入力モデルへ含めてApplication層へ渡す。

認証済みのユーザーが対象データを参照または操作できるかを判断する業務上の認可は、Application層またはDomain層の責務とする。

認可判断を含む場合でも、業務上の状態を変更しなければQueryとして扱う。

## 依存可能なレイヤー
- Domain層
- Python標準ライブラリ
- Pydantic

Infrastructure層やPresentation層の具体的な実装には依存しない。外部システムとの接続が必要な場合は、Application層またはDomain層で定義されたProtocolを利用する。

## 配置するもの
- 入力ポート
- 出力ポート
- ユースケース
- ユースケースの入力モデル
- ユースケースの出力モデル
- ユースケース固有の例外

## 配置しないもの
- HTTPステータスコードやHTTPレスポンスの生成
- FastAPIの`Request`、`Response`、`Depends`
- データベース、LLM、S3などの具体的な接続処理
- Domain層が保証すべきビジネスルール

## エラーハンドリング
- 入力モデルの構築時に構造的な不正がある場合は、Pydanticの`ValidationError`を発生させる
- ユースケース固有の業務チェックに違反した場合は、Application層で定義した例外を発生させる
- Infrastructure層のSDK固有例外をApplication層へ漏らさない

発生した例外をHTTPレスポンスなどに変換するのはその上の層のPresentation層の役割である。
Application層では自らが関係しない事柄に合わせたエラー変換はしないこと。

### Domain層からの例外
Domain層から送出された例外は、原則としてApplication層では捕捉せず、そのまま上位層へ送出する。

Domain例外は業務ルール違反を表しており、Application層でも意味のある例外である。そのため、同じ意味を持つApplication例外への不要な変換は行わない。

ただし、以下の場合はApplication層でDomain例外を捕捉してよい。

- 複数のDomain例外を、ユースケース上では同じ意味の例外として扱う場合
- ユースケース上の文脈を追加する必要がある場合
- 例外発生時に補償処理などを実行する必要がある場合

Domain例外を捕捉する場合も、HTTPステータスコードなどPresentation層の都合を持つ例外には変換しない。また、予期しないプログラムエラーをDomain例外やApplication例外へ一律に変換しない。

### Infrastructure層からの例外
Application層は、出力ポートの契約で定義された例外のみを扱う。Infrastructure層で利用するSDK、DBドライバー、外部APIクライアントなどの実装固有例外には依存しない。

出力ポートが送出する可能性のある例外は、呼び出し側が後続処理を判断できる粒度で、ポートを所有するApplication層またはDomain層に定義する。

例えば、一意制約違反を表すDB固有例外ではなく、対象が既に存在することを表す例外として定義する。Infrastructure層は、実装固有例外をこの契約で定義された例外へ変換する責務を持つ。

Application層は、出力ポートから受け取った例外について、以下のいずれかを行う。

- ユースケースとして代替処理や補償処理を実行する
- ユースケース上の意味を持つApplication例外へ変換する
- Application層で対応する必要がない場合は、そのまま上位層へ送出する

出力ポートから受け取った例外を、HTTPステータスコードなどPresentation層の都合を持つ例外へ変換しない。

## テスト方針
- ユースケースの正常系、代替系、異常系をテストする
- 入力モデルの制約は、同値分割と境界値分析に基づいてテストする
- Repositoryや外部サービスへの呼び出しは、出力ポートのモックまたはスタブに差し替える
- トランザクションは、正常終了時にコミットされ、例外発生時にロールバックされることをテストする
- LLM呼び出しはモックまたはスタブに差し替え、出力のスキーマと必要なキーワードを検証する
