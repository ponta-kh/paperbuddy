# バックエンドAPI共通規約

HTTP API全体で共通して適用する外部公開形式を定義する。

## URL
- URLにはリソースを表す名詞を使用する
- リソース名は複数形とする
- URLに処理を表す動詞を原則として含めない
- URLの単語区切りにはハイフンを使用する

```text
GET /chats/{chat_id}
POST /chats
```

## HTTPメソッド
- `GET`: リソースを取得する
- `POST`: リソースを作成する、または単純なCRUDで表現できない処理を実行する
- `PUT`: リソース全体を置き換える
- `PATCH`: リソースの一部を更新する
- `DELETE`: リソースを削除する

## パラメータ
- リソースを識別する値はパスパラメータとして受け取る
- 絞り込み、並び順、ページネーションなどの取得条件はクエリパラメータとして受け取る
- 作成・更新する内容や複雑な入力値はリクエストBodyとして受け取る
- 認証情報は、採用する認証方式で定められたHTTPヘッダーまたはCookieから受け取る

## ステータスコード
- 取得成功は`200 OK`とする
- 作成成功は`201 Created`とする
- レスポンスBodyを返さない成功は`204 No Content`とする
- JSON構文など、リクエストを解析できない場合は`400 Bad Request`とする
- 認証されていない場合は`401 Unauthorized`とする
- 業務上の認可がない場合は`403 Forbidden`とする
- 対象リソースが存在しない場合は`404 Not Found`とする
- 現在のリソース状態と競合する場合は`409 Conflict`とする
- リクエストを解析できるが、型や制約に違反している場合は`422 Unprocessable Entity`とする
- 予期しないサーバー内部の障害は`500 Internal Server Error`とする

## データ表現
- JSONのフィールド名は`snake_case`とする
- 日時はタイムゾーンを含むISO 8601形式で表現する
- UUIDはハイフンを含む標準文字列表現とする
- 列挙値は文字列で表現する
- 金額など浮動小数点数で誤差が問題になる値は、浮動小数点数で表現しない

## エラーレスポンス
エラーレスポンスは、クライアントがエラーの種類を判別できる機械可読な`code`と、利用者向けの`message`を含める。

```json
{
  "code": "chat_not_found",
  "message": "指定されたチャットは存在しません"
}
```

- `code`はAPI内で一意かつ安定した値とする
- `message`はPresentation層で定義した利用者向けの文言とする
- Domain例外およびApplication例外のメッセージを、そのまま`message`として使用しない
- `message`にスタックトレース、SDK固有エラー、DB情報などの内部情報を含めない
- 入力項目ごとのエラー詳細が必要な場合は、共通形式を定義して追加する

## 未決定事項
以下は、必要となった時点で共通形式を定義する。

- APIバージョニング
- ページネーション
- 並び順と絞り込み条件
- 冪等性キー
- レート制限
