# バックエンド機能開発

リポジトリの最新アーキテクチャと各層の責務に従い、バックエンドの新規機能を実装する。

## 必須の参照資料

リポジトリ内のドキュメントを正とする。すべてのパスはリポジトリルートから解決する。

- 作業開始時、ファイル配置や依存方向を決定する前、最終確認前に `docs/backend/architecture.md` を読み直す。
- 影響する各層の計画または編集を始める直前に、`docs/backend/layer/` 配下の対応ファイルを読み直す。
  - Domain層: `docs/backend/layer/domain.md`
  - Application層: `docs/backend/layer/application.md`
  - Infrastructure層: `docs/backend/layer/infrastructure.md`
  - Presentation層: `docs/backend/layer/presentation.md`
  - Dependencies層: `docs/backend/layer/dependencies.md`
- `docs/backend/specification/README.md` を読み、仕様書の配置規則を確認する。
- 対象ユースケースの仕様は `docs/backend/specification/usecases/[domain名]/` 配下から特定し、実装前、テスト計画前、最終確認前に読み直す。
- 対象ドメインの仕様書が複数ある場合は、変更対象のユースケースに関係する仕様書をすべて確認する。
- 対象ユースケースの仕様書が存在しない、または要求と矛盾する場合は、推測で実装を進めずユーザーへ明示する。
- スコープが変わった場合、設計判断に迷った場合、実装中に矛盾を発見した場合は参照資料を読み直す。
- 過去に読んだ内容に依存せず、参照資料をスキル内へ複製しない。常にリポジトリ上の最新版を参照する。
- 各レイヤーの責務、I/O、エラーハンドリング、依存関係、実装規則は、対応する `docs/backend/layer/*.md` の記述に従う。
- 近接するバックエンドコードとテストを調査し、設計ドキュメントを具体的な実装へ落とし込む際の命名・記述・テスト規約を合わせる。
- ドキュメントと既存コードが矛盾する場合は、矛盾した実装パターンを拡大する前に明示する。

## 作業手順

### 1. スコープを確定する

1. `AGENTS.md`、`docs/backend/architecture.md`、バックエンドのパッケージ設定を読む。
2. `docs/backend/specification/usecases/[domain名]/` 配下から対象ユースケースの仕様書を特定して読む。
3. 既存のバックエンド構成と、最も類似する機能を調査する。
4. 要求された振る舞い、影響する層、外部依存、エラーケース、受け入れ条件を特定する。
5. 公開API、ドメインの振る舞い、永続化、外部サービスに影響する前提条件を明示する。

### 2. 層ごとに計画する

1. `docs/backend/architecture.md` を読み直す。
2. `docs/backend/specification/usecases/[domain名]/` 配下の対象ユースケース仕様書を読み直す。
3. `docs/backend/layer/` 配下から、影響するすべての層のドキュメントを読み直す。
4. 各レイヤー資料から、責務、配置先、I/O、エラーハンドリング、許可される依存方向を整理する。
5. ユースケース仕様の各振る舞いを担当する層へ割り当て、必要なインターフェースと依存関係の配線を特定する。
6. 設計資料に記載された依存方向を維持する。
7. 仕様書の正常系、代替系、異常系、制約を基に、同値分割と境界値分析を用いてテストを計画する。

### 3. 層ごとに実装する

影響する各層について、以下を実施する。

1. 編集を始める直前に、その層のドキュメントを読み直す。
2. 類似する既存ファイルとテストを調査する。
3. その層の責務、I/O、エラーハンドリング、依存規則を満たすことを確認する。
4. 機能要件を満たす、最小限で一貫した変更を行う。
5. 層の境界を維持し、設計ドキュメントで定義された方法で層間を接続する。

Dependencies層を変更する場合も、`docs/backend/layer/dependencies.md` に記載された責務、依存関係の組み立て方、ライフサイクル、エラーハンドリング、テスト方針に従う。ほかの層と同様に、編集直前と最終確認時に資料を読み直す。

リポジトリのコマンド規約に従う。

- Pythonでは `uv add`、`uv run`、`uv sync` を使用する。`pip` は使用しない。
- Node.jsでは `pnpm add` と `pnpm install` を使用する。`npm` と `yarn` は使用しない。
- APIキーやAWSクレデンシャルをコードに直書きしない。

### 4. テストする

振る舞いを所有する層にテストを追加または更新する。

- 代表的な同値クラスと境界値を網羅する。
- 該当する場合は、正常系、バリデーションエラー、Domain層・Application層のエラー、アダプター障害をテストする。
- ユニットテストではInfrastructure層をモックまたはスタブへ置き換える。
- Dependencies層のテストは `docs/backend/layer/dependencies.md` に記載された方針と、依存関係の配線によるリスクに従って設計する。
- LLM出力は文章の完全一致ではなく、スキーマ検証と必須キーワード検証を行う。
- 最初に対象を絞った関連テストを実行し、利用可能な場合はバックエンド全体のテストを実行する。

### 5. 最終確認する

1. `docs/backend/architecture.md` と影響したすべての層のドキュメントを読み直す。Dependencies層を変更した場合は `docs/backend/layer/dependencies.md` も対象に含める。
2. `docs/backend/specification/usecases/[domain名]/` 配下の対象ユースケース仕様書を読み直す。
3. 最終的な振る舞い、各層の責務、ファイル配置、依存方向、I/O型、エラーハンドリングを仕様書および設計ドキュメントと比較する。
4. バックエンドプロジェクトに設定されたフォーマット、Lint、型チェック、テストを実行する。
5. 変更した振る舞い、影響した層、実施した検証、残っている曖昧さやリスクを報告する。

## 制約

- ファイル数を減らす目的だけで、責務を別の層へ移動しない。
- アーキテクチャ、外部境界、または意味のある再利用に必要でない抽象化を追加しない。
- ユーザーの要求を超えて、機能のスコープを無断で拡大しない。
- ユーザーが設計変更を明示的に依頼していない限り、機能開発の副作用としてアーキテクチャドキュメントを変更しない。
- 作業ツリーに存在する無関係なユーザー変更を維持する。
