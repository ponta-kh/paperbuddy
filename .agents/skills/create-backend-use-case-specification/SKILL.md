# バックエンドユースケース仕様書作成

バックエンド機能を実装する前に、ユースケース単位の仕様書を作成する。

## 出力規則

- 仕様書は `docs/backend/specification/usecases/[domain名]/[use_case名].md` に作成する。
- `[domain名]` と `[use_case名]` は、既存コードおよびドキュメントの命名規則に合わせた英小文字のsnake_caseとする。
- 1つの仕様書には1つのユースケースのみを記載する。
- 雛形として `assets/use-case-specification-template.md` を使用する。
  - 雛形を必ずベースとしてコピーし、テンプレート項目は削除せず内容を記入するか「該当なし」と記載する。
  - テンプレートに項目を追加することは可能だが、既存項目の削除は許可しない。
  - テンプレートが存在しない場合は、`未確定事項` に「テンプレート不在」と記載し、最低限の必須セクション(目的、入出力、正常系、異常系、制約、受け入れ条件) を含むファイルを作成する。
- 雛形の項目が対象ユースケースに該当しない場合は、項目を削除せず「該当なし」と理由を記載する。
- 未確定事項を推測で確定せず、`未確定事項` セクションへ記載する。
- 基本フローにはMermaidの`flowchart TD`を使用し、処理、分岐、代替系、異常系を可視化する。
- Mermaid内の例外終端には具体的な例外型を記載し、例外メッセージは記載しない。
- ユースケース仕様書には、Application層とDomain層が関知する振る舞い、契約、業務上の制約のみを記載する。
- HTTPエンドポイント、HTTPメソッド、リクエストBody、HTTPステータスコード、公開エラーコードなど、Presentation層固有の情報を記載しない。
- SDK固有型、通信形式、接続設定、リトライ方式、現在利用する具体的な外部サービス名など、Infrastructure層の実装詳細を記載しない。外部依存は、Application層が必要とする能力を表す抽象的な名称で記載する。
- 出力ポートのProtocol名、操作名、契約例外名は、現在の外部サービス、製品、SDK、DBなどの実装に引っ張られず、Application層またはDomain層が必要とする能力と失敗理由を表す抽象的な名称にする。

## 必須の参照資料

リポジトリ内の最新ドキュメントを正とする。

- 作業開始時と最終確認前に `docs/backend/architecture.md` を読む。
- `docs/backend/specification/README.md` を読み、仕様書の配置規則に従う。
- `docs/backend/layer/` 配下のすべてのレイヤー資料を読み、仕様が各層の責務、I/O、エラーハンドリング、依存関係と矛盾しないことを確認する。
  - Domain層: `docs/backend/layer/domain.md`
  - Application層: `docs/backend/layer/application.md`
  - Infrastructure層: `docs/backend/layer/infrastructure.md`
  - Presentation層: `docs/backend/layer/presentation.md`
  - Dependencies層: `docs/backend/layer/dependencies.md`
- `docs/backend/specification/usecases/[domain名]/` が存在する場合は、同じドメインの既存ユースケース仕様書を読み、用語、粒度、命名、記述形式を合わせる。
- `docs/backend/specification/domain/[domain名]/domain.md` が存在する場合は読み、ドメインルール、Aggregate、不変条件、Domain例外と矛盾しないことを確認する。
- `docs/backend/specification/infrastructure/` 配下に関連する外部接続仕様書が存在する場合は読み、外部依存、例外、制約と矛盾しないことを確認する。
- 既存実装が存在する場合は、関連コードとテストを調査する。ただし、既存実装だけを根拠に新しい要求を推測しない。
- ドキュメント、既存仕様、ユーザー要求が矛盾する場合は、仕様書上で矛盾と判断が必要な点を明示する。

## 作業手順

### 1. 対象を特定する

1. ユーザー要求からドメイン名とユースケース名を特定する。
2. アクター、目的、開始条件、完了条件を整理する。
3. 対象ユースケースに関係する既存仕様書、アーキテクチャ資料、すべてのレイヤー資料、コード、テストを調査する。
4. 情報が不足していても安全に仮置きできる内容と、確認が必要な内容を分ける。
5. 関連するドメイン仕様書や外部接続仕様書が不足している場合は記録する。ただし、明示的な依頼なしに別種の仕様書を作成しない。

### 2. 仕様を整理する

以下を明確にする。

- ユースケースの目的と対象範囲。
- アクターと事前条件。
- 入力、出力、およびそれぞれの制約。
- 正常系の基本フロー。
- 代替系と異常系。
- ドメインルール、認可、整合性、冪等性、トランザクションなどの制約。
- 抽象化された外部依存、永続化、イベントなどの副作用。
- 外部依存へ要求する能力と、ユースケースが区別する必要がある失敗理由。具体的な実装製品名を出力ポートの契約名に使用しない。
- 受け入れ条件とテスト観点。
- 未確定事項と対象外。

### ユースケース分割の判断基準

入力または出力に複数の形が存在することだけを理由に、機械的にユースケースを分割しない。

以下が異なる場合は、ユースケースの分割を検討する。

- アクターが達成しようとする目的。
- 認可要件。
- トランザクション境界。
- 発生する副作用。
- 正常終了として保証する事後条件。

分割しない場合は、複数の入力・出力パターンを1つのユースケースとして扱う理由を仕様書へ明記する。業務判断をPresentation層へ移動してユースケースを分割してはならない。

### 3. 仕様書を作成する

1. `assets/use-case-specification-template.md` を基に仕様書を作成する。
2. `docs/backend/specification/usecases/[domain名]/[use_case名].md` へ配置する。
3. 実装方法ではなく、外部から観測可能な振る舞いと業務上の制約を中心に記述する。
4. 実装上の層やコンポーネントへ言及する場合は、対応するレイヤー資料に記載された責務、I/O、エラーハンドリング、依存関係に従う。
5. 正常系、代替系、異常系は、実装者とテスト作成者が同じ解釈をできる具体性で記述する。
6. Domainルールは`[DR-XX@ドメイン名]`、ユースケース固有のビジネスルールは`[BR-XX]`で参照できるようにする。
7. Domain例外、Application例外、出力ポートの例外は具体的な例外型で記載する。
8. トランザクション境界が、Application層の「1ユースケースを1トランザクション境界」と矛盾しないことを確認する。
9. 出力ポートのProtocol名、操作名、例外名に、特定の外部サービス、製品、SDK、DBなどの実装名が含まれていないことを確認する。
10. Presentation層またはInfrastructure層だけが関知する情報を、ユースケース仕様として記載していないことを確認する。

### 4. 最終確認する

1. `docs/backend/architecture.md` と `docs/backend/layer/` 配下のすべてのレイヤー資料を読み直す。
2. 同じドメインの既存仕様書と用語・粒度・命名を比較する。
3. 入出力、フロー、エラー、制約、受け入れ条件の間に矛盾がないことを確認する。
4. 同値分割と境界値分析に必要な情報が仕様書から読み取れることを確認する。
5. 異常系ごとに、具体的な例外、副作用、ロールバック、呼び出し元への結果が明確であることを確認する。
6. 出力ポートと契約例外の名称が、実装ではなく必要な能力と失敗理由を表していることを確認する。
7. HTTPやSDKなど、ユースケースの依存範囲外の情報が記載されていないことを確認する。
8. 作成したファイル、確定した内容、未確定事項を報告する。
9. 関連するドメイン仕様書または外部接続仕様書が不足している場合は報告する。

## 制約

- ユーザー要求にない機能を、仕様として無断で追加しない。
- 技術的な実装詳細だけでユースケースの振る舞いを定義しない。
- 出力ポートと契約例外を、特定のInfrastructure実装に依存する名称で定義しない。
- Presentation層またはInfrastructure層だけが関知する情報をユースケース仕様書へ持ち込まない。
- 未確定事項や矛盾を隠さない。
- 既存の仕様書を、明示的な依頼なしに変更しない。
- 暫定雛形を正式な標準として扱わない。正式な雛形が決定した場合は差し替える。
