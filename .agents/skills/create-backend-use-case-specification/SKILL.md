# バックエンドユースケース仕様書作成

バックエンド機能を実装する前に、ユースケース単位の仕様書を作成する。

## 出力規則

- 仕様書は `docs/backend/specification/[domain名]/[use_case名].md` に作成する。
- `[domain名]` と `[use_case名]` は、既存コードおよびドキュメントの命名規則に合わせた英小文字のsnake_caseを基本とする。
- 1つの仕様書には1つのユースケースのみを記載する。
- 暫定雛形として `assets/use-case-specification-template.md` を使用する。
- 雛形の項目が対象ユースケースに該当しない場合は、項目を削除せず「該当なし」と理由を記載する。
- 未確定事項を推測で確定せず、`未確定事項` セクションへ記載する。

## 必須の参照資料

リポジトリ内の最新ドキュメントを正とする。

- 作業開始時と最終確認前に `docs/backend/architecture.md` を読む。
- `docs/backend/layer/` 配下のすべてのレイヤー資料を読み、仕様が各層の責務、I/O、エラーハンドリング、依存関係と矛盾しないことを確認する。
  - Domain層: `docs/backend/layer/domain.md`
  - Application層: `docs/backend/layer/application.md`
  - Infrastructure層: `docs/backend/layer/infrastructure.md`
  - Presentation層: `docs/backend/layer/presentation.md`
  - Dependencies層: `docs/backend/layer/dependencies.md`
- `docs/backend/specification/[domain名]/` が存在する場合は、同じドメインの既存仕様書を読み、用語、粒度、命名、記述形式を合わせる。
- 既存実装が存在する場合は、関連コードとテストを調査する。ただし、既存実装だけを根拠に新しい要求を推測しない。
- ドキュメント、既存仕様、ユーザー要求が矛盾する場合は、仕様書上で矛盾と判断が必要な点を明示する。

## 作業手順

### 1. 対象を特定する

1. ユーザー要求からドメイン名とユースケース名を特定する。
2. アクター、目的、開始条件、完了条件を整理する。
3. 対象ユースケースに関係する既存仕様書、アーキテクチャ資料、すべてのレイヤー資料、コード、テストを調査する。
4. 情報が不足していても安全に仮置きできる内容と、確認が必要な内容を分ける。

### 2. 仕様を整理する

以下を明確にする。

- ユースケースの目的と対象範囲。
- アクターと事前条件。
- 入力、出力、およびそれぞれの制約。
- 正常系の基本フロー。
- 代替系と異常系。
- ドメインルール、認可、整合性、冪等性、トランザクションなどの制約。
- 外部サービス、永続化、イベントなどの副作用。
- 受け入れ条件とテスト観点。
- 未確定事項と対象外。

### 3. 仕様書を作成する

1. `assets/use-case-specification-template.md` を基に仕様書を作成する。
2. `docs/backend/specification/[domain名]/[use_case名].md` へ配置する。
3. 実装方法ではなく、外部から観測可能な振る舞いと業務上の制約を中心に記述する。
4. 実装上の層やコンポーネントへ言及する場合は、対応するレイヤー資料に記載された責務、I/O、エラーハンドリング、依存関係に従う。
5. 正常系、代替系、異常系は、実装者とテスト作成者が同じ解釈をできる具体性で記述する。

### 4. 最終確認する

1. `docs/backend/architecture.md` と `docs/backend/layer/` 配下のすべてのレイヤー資料を読み直す。
2. 同じドメインの既存仕様書と用語・粒度・命名を比較する。
3. 入出力、フロー、エラー、制約、受け入れ条件の間に矛盾がないことを確認する。
4. 同値分割と境界値分析に必要な情報が仕様書から読み取れることを確認する。
5. 作成したファイル、確定した内容、未確定事項を報告する。

## 制約

- ユーザー要求にない機能を、仕様として無断で追加しない。
- 技術的な実装詳細だけでユースケースの振る舞いを定義しない。
- 未確定事項や矛盾を隠さない。
- 既存の仕様書を、明示的な依頼なしに変更しない。
- 暫定雛形を正式な標準として扱わない。正式な雛形が決定した場合は差し替える。
